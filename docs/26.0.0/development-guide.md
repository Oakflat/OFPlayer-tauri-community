# OFPlayer Tauri v26.0.0 开发者指南

## 环境要求

- **Node.js**：18+
- **Rust**：最新 stable（通过 rustup 安装）
- **Tauri 前置依赖**：https://tauri.app/start/prerequisites/
  - Windows: WebView2、Visual Studio Build Tools

## 开发命令

### 前端（在 `ofplayer-tauri/` 执行）

```bash
npm install              # 安装依赖
npm run dev              # 启动 Vite 开发服务器 (port 5173)
npm run build            # 构建前端产物到 dist/
```

### Tauri 开发（在 `ofplayer-tauri/` 执行）

```bash
npm run tauri:dev        # 启动 Tauri 开发模式（前端 + Rust 后端）
npm run tauri:build      # 构建完整安装包
```

### Rust 后端（在 `src-tauri/` 执行）

```bash
cargo fmt                # 格式化代码
cargo check              # 类型检查
cargo test               # 运行测试
```

## 项目架构快速理解

### 前端数据流

```
App.vue (模板 + 事件)
  ↓ emit
ofplayerApp.js (编排逻辑)
  ↓ 调用
services/* (数据服务)
  ↓ invoke
desktopDataService.js (Tauri IPC 封装)
  ↓ invoke
Rust lib.rs (#[tauri::command])
  ↓
desktop_state.rs / playback.rs / storage.rs
  ↓
SQLite
```

### 后端命令注册流程

新增 Tauri command 需要修改以下位置：

1. **Rust 函数**：在 `lib.rs` 中添加 `#[tauri::command]` 函数
2. **Handler 注册**：在 `tauri::generate_handler![...]` 中添加函数名
3. **构建脚本**：在 `build.rs` 的 `APP_COMMANDS` 数组中添加函数名
4. **权限生成**：运行 `cargo check` 自动生成 `permissions/autogenerated/*.toml`
5. **能力配置**：在 `capabilities/default.json`（或其他窗口 capability）中添加权限
6. **前端调用**：在 `services/desktopDataService.js` 中添加 `invoke()` 封装
7. **类型定义**：如返回复杂结构，在 `desktop_types.rs` 中定义类型
8. **验证**：`cargo check` 确认权限生成和 command 注册正确

### 前端 Service 层

```js
// services/desktopDataService.js 中的典型封装
async function loadBootstrap(options) {
  return invoke('desktop_state_load_bootstrap', options)
}

async function selectTrack(trackId, queueTrackIds) {
  return invoke('playback_session_select_track', {
    request: { trackId, queueTrackIds }
  })
}
```

### Store 层

Store 使用 Vue 3 `ref()` / `computed()` 实现响应式状态：

```js
// stores/libraryStore.js
const tracks = ref([])
const libraries = ref([])

function getTrackById(id) {
  return tracks.value.find(t => t.id === id) ?? null
}
```

## 关键设计原则

### 启动路径

- Bootstrap 只传 ID、计数、偏好、状态、摘要
- 不传完整 tracks、artwork base64、歌词全文
- 完整数据按需懒加载

### 曲库封面和远端元数据

- `tracks.payload_json` 不存封面 base64；这是性能红线。
- 封面必须走 `track_artwork` 和本地 `track-artwork/` 资产目录。
- WebDAV 同步只做索引，不为全库下载音频来解析封面。
- WebDAV 封面补齐只能在播放缓存或显式单曲补齐后进行，并由后端托管为本地资产。
- 评审涉及 catalog / metadata / WebDAV 的改动时，先读 `docs/catalog-artwork-storage.md`。

### 歌词胶囊

- 主窗口只 open/close/toggle，不转发数据
- 胶囊窗口主动 boot，Rust 直连
- 进度本地插值，关键动作才发 anchor
- 音浪由独立 meter emitter 发送

### 托管存储

- 必须带 `.ofplayer-managed-storage.json` marker
- 清理前校验 marker
- 不碰用户原始扫描目录

## 调试

### 诊断日志

位置：`C:\Users\<user>\AppData\Local\ofplayer\diagnostics\ofplayer-diagnostics.ndjson`

格式：每行一个 JSON 对象，包含 `timestamp`、`level`、`category`、`event`、`payload`。

关键事件：
- `session_started` / `tauri_setup`
- `bootstrap_state_loaded` / `bootstrap_snapshot` / `bootstrap_state_ready`
- `startup_splash_hidden` / `app_startup`
- `playback_session_select_track_profile`
- `capsule_diagnostics_summary`
- `storage_gc` / `storage_gc_failed`

### SQLite 检查

```sql
-- 查看最大的 app_state 条目
SELECT key, length(value_json)
FROM app_state
ORDER BY length(value_json) DESC
LIMIT 10;

-- 确认旧缓存已清除
SELECT count(*)
FROM app_state
WHERE key = 'snapshot.catalog';
-- 应返回 0
```

### Vite 开发服务器

- 主窗口：`http://localhost:5173`
- 胶囊窗口：`http://localhost:5173/lyric-capsule.html`

开发模式下，Rust `println!` / `eprintln!` 输出到终端。

## 常见问题

### 开发服务器启动失败

检查端口 5173 是否被占用。Vite 配置了 `strictPort: true`。

### Rust 编译错误

运行 `cargo check` 查看详细错误。常见原因：
- 新增 command 未在 `generate_handler!` 中注册
- 权限 TOML 未更新（运行 `cargo check` 自动重新生成）
- `capabilities/*.json` 中缺少新权限

### WebView2 问题

Windows 需要 WebView2 运行时。Windows 11 自带，Windows 10 需要安装。

### CSS 样式不生效

Vite 配置了 `cssCodeSplit: false` 和 `cssMinify: false`。WebView2 中 `backdrop-filter` 需要保留两个声明。

## 代码风格

- 前端：遵循 Vue 3 Composition API 风格，使用 `ref()`、`computed()`、`watch()`
- 后端：遵循 Rust 标准风格，使用 `cargo fmt` 格式化
- 提交信息：使用约定式提交（feat/fix/perf/ui/docs/chore/refactor）
